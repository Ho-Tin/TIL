# 삼성청년 SW·AI 아카데미: 관통 프로젝트 06 요약

이 문서는 '삼성청년 SW·AI 아카데미'의 '관통 프로젝트 06' 관련 강의 내용을 요약한 것입니다. 주요 내용은 **데이터베이스 모델링**, **정규화**, 그리고 이를 적용한 **Django 레시피 관리 프로젝트 실습**입니다.

## 1. 🗃️ 데이터베이스 모델링

데이터베이스를 효율적이고 일관성 있게 사용하기 위한 설계 과정입니다.

### 모델링의 중요성 (00:03)

* **효율성:** 잘 설계된 구조는 쿼리 성능과 저장 효율을 향상시킵니다.
* **일관성:** 데이터의 중복과 이상 현상을 최소화하여 모순이 없도록 합니다.
* **무결성 보장:** 제약 조건을 통해 부적절한 데이터의 입력을 방지합니다.

### 무결성 (Integrity) (00:04 ~ 00:07)

데이터의 일관성과 신뢰성을 유지하는 것이며, 주요 제약 조건은 다음과 같습니다.

* **개체 무결성 (Entity Integrity):**
    * 테이블의 **기본 키(Primary Key, PK)**는 **UNIQUE**해야 하며 **NULL** 값을 가질 수 없습니다.
* **도메인 무결성 (Domain Integrity):**
    * 컬럼의 값은 정의된 **데이터 타입, 형식, 범위**를 벗어날 수 없습니다.
    * 예: `CHECK (price >= 0)`
* **기타 제약 조건:**
    * **고유성 (UNIQUE):** PK는 아니지만 중복된 값을 허용하지 않습니다. (NULL은 허용될 수 있음)
    * **NULL 무결성 (NOT NULL):** 특정 컬럼이 NULL 값을 가질 수 없도록 강제합니다.
    * **일반 무결성 (General Integrity):** 비즈니스 로직에 따른 제약 (예: '잔고는 0 미만이 될 수 없다')

### 모델링 진행 절차 (00:08)

1.  **요구사항 수집 및 분석**
2.  **개념적 설계** (ERD 작성 등)
3.  **논리적 설계** (테이블 스키마 정의)
4.  **물리적 설계** (실제 DB에 구현)

## 2. ✨ 데이터베이스 정규화 (Normalization)

데이터의 중복을 줄이고 이상 현상(Anomaly)을 방지하기 위해 테이블을 분해하는 과정입니다.

### 정규화의 목적 (00:15)

1.  **중복 최소화:** 불필요한 데이터 중복을 제거하여 일관성을 유지합니다.
2.  **이상 현상 방지:** 데이터 삽입, 갱신, 삭제 시 발생할 수 있는 문제를 예방합니다.
3.  **유연성 향상:** 데이터베이스 구조 변경의 영향을 최소화하고 유지보수성을 높입니다.

### 이상 현상 (Anomaly) (00:16 ~ 00:19)

* **갱신 이상 (Update Anomaly):** 중복된 데이터 중 일부만 갱신되어 데이터 불일치가 발생합니다. (예: 재료 '토마토'가 여러 레시피에 텍스트로 저장된 경우)
* **삭제 이상 (Deletion Anomaly):** 특정 데이터를 삭제할 때, 원하지 않았던 다른 정보까지 함께 삭제됩니다. (예: '명란 파스타' 레시피를 삭제했더니 '명란'이라는 재료 정보가 DB에서 완전히 사라짐)
* **삽입 이상 (Insertion Anomaly):** (언급됨) 불필요한 데이터가 없으면 새 데이터를 삽입할 수 없습니다.

### 정규화 과정 (00:21 ~ 00:32)

* **제1정규형 (1NF):** (00:23)
    * **정의:** 테이블의 모든 컬럼이 **원자값(Atomic Value)**을 가져야 합니다.
    * **예:** '전화번호' 컬럼에 '010-1111-2222, 010-3333-4444'처럼 여러 값을 저장하면 안 됩니다.
* **제2정규형 (2NF):** (00:25)
    * **정의:** 1NF를 만족하고, **부분 함수 종속**을 제거해야 합니다. (복합 키의 일부에만 종속되는 속성 분리)
* **제3정규형 (3NF):** (00:27)
    * **정의:** 2NF를 만족하고, **이행 함수 종속**을 제거해야 합니다. (기본 키가 아닌 속성이 다른 속성을 결정하는 경우)
    * **예:** `학생번호` → `학과` → `학과장`. '학과장'은 '학과'에 종속되므로 별도 테이블로 분리합니다.
* **BCNF (Boyce-Codd Normal Form):** (00:30)
    * **정의:** 3NF보다 엄격하며, 모든 **결정자**가 **후보 키**여야 합니다.

## 3. 🧑‍🍳 레시피 관리 프로젝트 (Django 실습)

정규화 원칙을 Django 프로젝트에 적용하는 실습입니다.

### N:M 관계 모델링 (00:20, 00:34)

* **문제:** 하나의 **레시피**는 여러 **재료**를 갖고, 하나의 **재료**는 여러 **레시피**에 사용될 수 있습니다. (다대다, N:M 관계)
* **해결:** '이상 현상'을 방지하기 위해 `Recipe` 테이블과 `Ingredient` 테이블을 분리하고, 이 둘을 연결하는 **중간 테이블(Junction Table)**을 둡니다.
* **Django 적용:** `Recipe` 모델에 `ingredients = models.ManyToManyField(Ingredient)` 필드를 추가합니다.

### Django Model, Form, View (00:34 ~ 00:45)

* **Model (00:34, 00:37):**
    * `Recipe` 모델과 `Ingredient` 모델을 정의합니다.
    * `ManyToManyField`를 사용해 N:M 관계를 설정합니다.
* **Migration (00:39):**
    * `python manage.py makemigrations`
    * `python manage.py migrate`
    * 위 명령을 실행하면 Django가 `foods_recipe`, `foods_ingredient`, 그리고 중간 테이블인 `foods_recipe_ingredients`를 자동으로 생성해줍니다.
* **Form (00:41):**
    * `forms.ModelForm`을 상속받습니다.
    * `ingredients` 필드에 `forms.ModelMultipleChoiceField` 위젯(주로 체크박스)을 사용하여 여러 재료를 한 번에 선택할 수 있게 합니다.
* **View & Template (00:42, 00:44, 00:45):**
    * **생성 (Create):** `ModelForm`을 사용하면 `form.save()` 메서드 하나로 `Recipe` 객체 생성과 N:M 관계(중간 테이블) 데이터 저장이 **자동으로 처리**됩니다.
    * **조회 (Read):** 템플릿에서 `recipe.ingredients.all`처럼 `.`(dot) 연산자를 통해 N:M 관계에 있는 재료들을 쉽게 조회하고 반복문(`{% for %}`)으로 출력할 수 있습니다.

### 생성 결과 (00:43)


* 데이터베이스에 `foods_recipe` (레시피), `foods_ingredient` (재료), `foods_recipe_ingredients` (중간 테이블) 3개의 테이블이 생성되어 정규화된 데이터 구조를 확인할 수 있습니다.
